Order Imbalance（订单不平衡）策略，本质上就是一句话：

看买方和卖方谁更强，然后赌价格在很短时间内会朝强的一边偏。

它是微观结构里最经典、最基础的一类策略。你前面一直在看的盘口不平衡、OFI、microprice、Tape-OFI，其实都可以看成是 Order Imbalance 体系 的不同实现。

1. Order Imbalance 到底是什么

在微观交易里，Order Imbalance 通常泛指：

买盘力量 - 卖盘力量 的不平衡。

最常见的表达是这种标准化比值：

Imbalance=
Buy Side+Sell Side
Buy Side−Sell Side
	​


这个值的含义很直观：

接近 +1：买方明显更强
接近 -1：卖方明显更强
接近 0：两边差不多

但这里的 “Buy Side / Sell Side” 可以有不同定义，所以 Order Imbalance 其实有好几种流派。

2. 它常见有 3 种
第一种：Order Book Imbalance

这是最常见的盘口不平衡。

比如只看买一卖一：

BidSize+AskSize
BidSize−AskSize
	​


也可以看前 3 档、前 5 档、前 10 档，再做加权。

它反映的是：

当前挂在盘口上的静态力量对比。

也就是：
“现在买盘堆得更厚，还是卖盘堆得更厚？”

第二种：Order Flow Imbalance（OFI）

这个不是看“现在挂了多少”，而是看：

盘口变化过程中，哪边更主动。

比如：

买一价格上升
买一挂单增加
卖一价格下降
卖一挂单减少

这些都说明买方更积极。

它反映的是：

挂单流入/撤单/价位变化导致的动态不平衡。

所以 OFI 比单纯盘口不平衡更“活”。

第三种：Trade Imbalance / Tape Imbalance

这是看逐笔成交。

比如最近 0.5 秒、1 秒、2 秒里：

主动买成交量 > 主动卖成交量
或者
打在 ask 的成交明显更多

这代表：

真正成交出来的主动性，哪边更强。

这个比盘口更真实，但也更“晚一步”。

3. 这个策略为什么有效

因为短线价格不是随机跳的，它经常是被下面几件事推着走：

1）一边队列更容易被打穿

如果卖一很薄、买盘很厚，那主动买单一来，价格更容易往上跳。

2）市场里很多人都在盯同样的信息

你看到买方更强，别的做市商、超短线、高频也可能看到。
于是他们会：

提高买价
撤掉便宜卖单
主动追买

价格就更容易朝那个方向偏。

3）microprice 会先偏

即使 mid 还没动，盘口结构已经在告诉你：

“下一跳更可能朝哪边”

这就是为什么 imbalance 常常和 microprice 一起用。

4. Order Imbalance 策略的核心思路

它不是“看到买盘大就无脑买”，而是：

第一步：判断不平衡是否足够明显

比如：

买盘是卖盘的 1.5 倍
或 imbalance > 0.2 / 0.3 / 0.4
或 z-score 超过阈值
第二步：判断这个不平衡是不是“真的”

因为盘口会骗人。
有些大单会撤掉，有些是冰山单，有些只是暂时挂着。

所以通常还要加确认条件，比如：

最近主动买成交也更强
OFI 同方向
microprice 向上偏
spread 仍然很窄
没有明显反向大单压着
第三步：决定怎么交易

这一步会分成两派：

A. Taker 派

直接冲进去。

逻辑是：

买方明显强
价格大概率马上上一个 tick
那我现在直接买，赚下一跳

这就是典型的超短线 scalp。

B. Maker 派

不急着追，而是调整挂单位置。

逻辑是：

买方明显强
那我更愿意挂买单，不愿意太积极地挂卖单
或者我把两边报价整体往上抬一点

这就是 skewed market making。

5. 它最经典的两种玩法
玩法 1：Order Imbalance 做 taker/scalp

这是你现在最关心的那类。

多头版本

当满足这些条件时买入：

多档盘口明显偏多
OFI 偏多
最近成交主动买更强
microprice 在 mid 上方
spread = 1 tick
对手盘较薄

然后：

主动买入
挂 1 tick 或 2 tick 止盈
如果流向反转，马上砍掉
空头版本

反过来做。

这种玩法的本质是：

抓“下一两跳”的延续。

它不是波段，也不是趋势交易，而是非常短的微观动量。

玩法 2：Order Imbalance 做 maker

这个更像做市。

当买方更强时

你会：

更积极挂 bid
更保守挂 ask
或整体报价上移
当卖方更强时

反过来：

更积极挂 ask
更保守挂 bid
或整体报价下移

它赚的不是纯方向，而是：

spread
rebate
短期价格偏移带来的报价优势

这种玩法对：

队列位置
撤单速度
费率结构
被反选风险

要求更高。

6. 它和 microprice、OFI、Tape-OFI 的关系

你可以这样理解：

Order Book Imbalance

看的是：
“现在谁堆得更厚”

OFI

看的是：
“谁在主动推动盘口变化”

Tape-OFI / Trade Imbalance

看的是：
“谁真的成交得更凶”

microprice

看的是：
“综合当前盘口后，短期价格更可能往哪边偏”

所以它们不是互斥关系，而是同一个大逻辑的不同层次。

一个很常见的组合是：

盘口不平衡 + OFI + Tape + microprice

也就是：

盘口告诉你谁强
OFI 告诉你谁更主动
Tape 告诉你谁真的打出来了
microprice 告诉你价格偏移方向

这套组合会比单独一个 imbalance 强很多。

7. 一个实战上的 Order Imbalance 模板

你可以把策略写成这种样子：

入场条件
spread 必须很窄，比如 1 tick
前 3~5 档加权 imbalance > 阈值
最近 0.5 秒 OFI 同方向
最近 0.5 秒主动成交同方向
microprice 偏向同方向
没有大新闻、停牌、异常跳价
做多

满足以上后：

直接 lift ask 买入
或者
在 bid 挂单抢队列
出场

满足任一条件就走：

盈利 1 tick / 2 ticks
imbalance 回落
OFI 翻转
Tape 翻转
microprice 回到中性
超时没走出来
库存/风险超限

这就是非常标准的微观结构订单不平衡策略。

8. 这个策略最适合什么市场

它最适合：

高流动性
低点差
盘口更新快
tick size 相对重要
短线订单流很密集

比如：

大盘活跃股
指数成分股
高频 ETF
活跃期货
高流动性加密永续

如果市场太慢、太稀、点差太大，这个策略会变差。

9. 哪些情况下最好用
1）大 tick 市场

当 tick size 相对大时，最优价位更有意义，队列更重要，盘口不平衡更有解释力。

2）高成交、高更新频率

这样信号更新快，确认更及时。

3）做超短

你持有时间越短，imbalance 的解释力越强。
时间拉长后，别的因素会压过它。

10. 哪些情况下容易失效
1）假挂单 / spoofing

盘口看起来买盘很厚，但下一秒撤掉了。

2）隐藏流动性

你以为对面很薄，结果对面有冰山单。

3）消息驱动跳价

新闻来的时候，盘口意义会瞬间变差。

4）延迟过高

你看到买盘强时，别人已经先动了。
你追进去，反而是接最后一棒。

5）只看静态盘口，不看动态流

这是最常见错误。
单纯盘口厚，不代表真的要涨。
所以 OFI 和 Tape 很重要。

11. Order Imbalance 的核心风险

这个策略最大的风险不是方向判断错，而是：

你以为有优势，其实只是看到了“表面优势”

也就是说：

盘口厚，但是假单
买盘多，但主动成交并不跟
OFI 强，但价格已经提前走完
你看见信号时，edge 已经被更快的人吃掉了

所以真正实战里，Order Imbalance 不能孤立用。

12. 怎么把它做得更强

最常见的增强方式有 6 个：

1）多档加权

不要只看买一卖一。
前 3 档、前 5 档通常更稳。

2）时间平滑

做 rolling mean / EWMA / z-score，减少噪声。

3）加入 OFI

从静态不平衡升级成动态不平衡。

4）加入 Tape

确认真实主动成交。

5）加入 microprice

看短期价格偏移是否一致。

6）做状态过滤

比如：

spread regime
volatility regime
event regime
time-of-day regime

这样误判会少很多。

13. 你可以把它分成 3 个层级理解
第一层：最原始版

只看盘口厚薄。

这是入门版，最简单，但也最容易被骗。

第二层：进阶版

盘口不平衡 + OFI + Tape。

这已经比较实战了。

第三层：专业版

盘口不平衡 + OFI + Tape + microprice + latency/queue model + inventory/risk。

这才接近真正可交易的微观策略。

14. 对你来说，最实用的结论

结合你一直在研究的方向，我会直接给你结论：

Order Imbalance 不应该被当成一个“单独策略”，而应该被当成一个核心信号层。

最好的用法不是：

“只要 imbalance > 0.3 我就买”

而是：

用 imbalance 判断主方向
用 OFI 判断主动性
用 Tape 判断真实成交
用 microprice 判断短期偏移
再决定是 maker 还是 taker
再配合 1 tick / 2 tick 的超短风控和止盈

这样它才会从一个“指标”，变成一个真正的微观交易系统。

15. 一句话总结

Order Imbalance 策略，本质上就是通过衡量买卖力量不平衡，去抓价格在极短时间里的方向偏移；最基础的是盘口不平衡，实战更强的是把它和 OFI、Tape、microprice、库存控制、延迟与队列模型一起用。

LOBCAST / lob-deep-learning 的 策略 思路全面讲解
Thought for 1m 8s

可以。先给你结论：

LOBCAST / lob-deep-learning 这类东西，本质上不是“写死规则的交易策略”，而是“用限价订单簿（LOB）做短周期价格方向预测”的深度学习框架。
它们的核心思路不是手工写“盘口不平衡 > 0.3 就买”，而是把过去一小段时间的盘口快照喂给模型，让模型自己从价格、挂单量、层级结构、时序变化里学出“未来几个 tick 的 mid-price 更可能上涨、下跌还是横盘”。LOBCAST 是一个更偏基准测试/训练评估框架的项目；lob-deep-learning 更偏具体模型实现仓库，实现了 DeepLOB、TransLOB、DeepFolio 等模型。(github.com
 )

一、它们到底在做什么

这类方法的目标，可以压缩成一句话：

用过去 T 个时刻的订单簿状态，预测未来短周期 mid-price 的方向。

LOBCAST 论文把这个任务叫做 SPTP（Stock Price Trend Prediction），也就是股票价格趋势预测；输入是一段时间窗口内的 LOB 观测，输出是一个三分类分布：上涨 U / 下跌 D / 稳定 S。LOBCAST 框架本身包含数据预处理、模型训练、测试评估和 profit analysis；论文还强调他们 benchmark 了 15 个深度学习模型，并发现这些模型在新数据上都会出现明显性能下降。(github.com
 )

所以它的“策略思路”不是：

先定义几个技术指标
再手工拼规则

而是：

把盘口序列变成模型输入
用监督学习学一个“下一段短期价格方向分类器”
再把分类结果转成交易信号 (arxiv.org
 )
二、为什么它们用 LOB，而不是只用 K 线或成交价

lob-deep-learning 的 README 讲得很直接：相比只用价格，LOB 是订单和成交信息的堆叠，信息量更大；而且时间尺度是毫秒级甚至事件级，作者认为这样更能捕捉超短周期微观结构，同时降低长期数据里那种“看起来有效、其实容易过拟合”的风险。(github.com
 )

直白讲，它赌的是这件事：

短线价格不是凭空动的，而是被盘口结构推动的。

也就是说，未来几跳往哪边走，不只是看“上一笔涨没涨”，而是看：

买一卖一谁更厚
前几档深度怎么分布
哪边撤单更快
哪边更容易被打穿
买卖盘层级结构有没有往某边倾斜

你前面一直在研究的 order imbalance、OFI、microprice，本质上都是这种思路的“手工版”；LOB deep learning 则是它的“自动特征学习版”。(github.com
 )

三、这类策略的完整逻辑链是什么

把它拆开看，其实就是 5 步。

1）先定义输入：过去一段盘口窗口

这类模型不是只看一个时刻的盘口，而是看一个时间窗口里的 LOB 序列。LOBCAST 论文里写得很清楚：给定一段时间窗口，提取这一小段市场观测作为输入；lob-deep-learning 的公开实验参数里，常用的是 T=100，也就是一次输入看 100 个时间步。(arxiv.org
 )

输入通常包含：

ask 各档价格
ask 各档数量
bid 各档价格
bid 各档数量

FI-2010 数据集是 10 档 LOB，并给了 10/20/30/50/100 tick 等不同预测 horizon；lob-deep-learning 还做了一个韩国期货 KRX 的 5 档版本。(github.com
 )

2）再定义标签：未来是涨、跌还是横盘

这是整个策略最关键的一步。

这类框架不是预测“未来精确价格”，而是预测一个短期方向标签。LOBCAST 论文把任务定义成 ternary classification：U、D、S。为了减少 mid-price 的高频噪声，不是直接拿相邻两个 mid-price 硬比较，而是把未来 horizon 里的价格做平滑平均，再跟当前价格比较，得到上涨/下跌/稳定标签。(arxiv.org
 )

lob-deep-learning 的 README 也给了类似标签定义：先算过去和未来窗口的平均价格，再根据一个阈值 α 把样本标成 Up / Down / Stationary。(github.com
 )

这一步其实已经把“交易思路”写进去了：

它不是做长线估值，也不是做日内大波段，而是做 very short horizon 的方向分类。

3）模型从盘口里自动学习“谁更强”

这一步才是 LOB deep learning 的核心。

以 lob-deep-learning 里的 DeepLOB/LOBster 为例，作者强调了一种 hierarchical feature integration 思路：

第一层卷积：整合买卖两边各自的价格 + 数量
第二层卷积：整合同一档位里的 ask/bid 对应关系
第三层卷积：整合 多档深度之间的结构
再用 inception / RNN（LSTM 或 GRU）去提取跨时间的模式 (github.com
 )

仓库作者甚至明确写了，这种分层整合能让网络学到价格和数量组合出来的有意义特征，例如 micro-price。(github.com
 )

这句话其实很关键，因为它说明：

这些模型学的，本质上还是你熟悉的微观结构信号，只不过不是你手写公式，而是模型自动抽取。

也就是说，模型很可能在隐式学习这些东西：

盘口不平衡
哪边挂单更厚
哪边更容易被吃穿
spread 状态
深度衰减形状
microprice 偏移
某些时间上的 queue depletion 模式 (github.com
 )
4）输出不是价格，而是一个方向概率

模型最终输出的是：

P(Up)
P(Stable)
P(Down)

也就是一个分类概率分布，而不是“未来价格一定涨 0.12%”。LOBCAST 论文明确说，SPTP predictor 的输出是 trend classes U / D / S 的概率分布。(arxiv.org
 )

这意味着它更像一个信号引擎，不是完整交易系统。

5）最后再把分类结果转成交易

真正实战时，一般会这样转：

如果 P(Up) 明显高于 P(Down)，做多
如果 P(Down) 明显高于 P(Up)，做空
如果 Stable 概率高，观望

然后再叠加：

置信度阈值
spread 过滤
成交量过滤
波动 regime 过滤
最大持仓时间
止盈止损 / time stop

LOBCAST 论文也提到他们的框架包含 profit analysis，而且贡献里明确写了 release 了一个 backtesting environment for profit analysis。这说明作者自己也知道：分类准确率不是终点，必须转成交易再看收益。(arxiv.org
 )

四、LOBCAST 和 lob-deep-learning 的区别
LOBCAST 更像“研究平台”

LOBCAST 的定位很明确：它是一个开源框架，用来做 LOB 数据预处理、训练、评估、profit analysis 和模型 benchmark；主线是看不同 deep learning 模型在 SPTP 任务上的鲁棒性和泛化能力。而且它不是只跑一个模型，而是 benchmark 了 15 个模型。(github.com
 )

所以 LOBCAST 的策略思路是：

“把短期方向预测当成标准任务，比较不同模型谁在这个任务上更强。” (arxiv.org
 )

lob-deep-learning 更像“模型样例仓库”

这个仓库更偏实现导向。首页就写了它实现了各种 LOB 深度学习模型，比如 DeepLOB、TransLOB、DeepFolio 等；README 里又重点展开了 DeepLOB 和作者自己的 LOBster。(github.com
 )

所以 lob-deep-learning 的策略思路是：

“把论文里的典型 LOB 模型实现出来，给你一个从数据、标签、模型到训练实验的最小可运行路径。” (github.com
 )

五、DeepLOB / LOBster 这类模型到底在“学什么”

这个问题最重要。

DeepLOB 的思想

DeepLOB 在这个仓库里的描述是：3 个卷积块 + inception 层 + LSTM。(github.com
 )

你可以把它理解成：

卷积块：像扫描仪一样，从盘口矩阵里抽局部结构特征
inception：多尺度看盘口模式
LSTM：再看这些模式在时间上的延续和变化 (github.com
 )

本质上它在试图回答：

“最近这一小段盘口演化，看起来更像马上上冲、下压，还是横着不动？”

LOBster 的思想

LOBster 是该仓库作者基于 KRX 5 档盘口做的轻量版。因为 5 档输入特征比 10 档少了一半，作者担心过拟合，于是用了 symmetric-mask dropout；同时把 LSTM 换成了 GRU。(github.com
 )

它的核心思路不是改交易哲学，而是：

在更浅的盘口、更高频的实际市场里，把 DeepLOB 做轻量化和适配。 (github.com
 )

六、把它翻译成交易员语言，它其实是什么策略

如果不用 AI 术语，而用交易语言来讲：

它不是“趋势跟踪”

因为 horizon 很短，通常是几十个 tick 量级。(github.com
 )

它也不是“基本面预测”

因为输入几乎全是盘口，不看财报、不看宏观。(arxiv.org
 )

可以。这个 limit-order-book-market-making 的策略思路，核心上其实很纯：

它不是预测涨跌的方向策略，而是一个典型的“库存中性、双边报价、赚 spread”的基础做市策略。
项目 README 直接写了两件事：一是它实现了一个 LOB 撮合引擎和一个 market maker agent；二是这个 agent 会围绕 mid-price 持续挂双边报价，目标是提供流动性并获得稳定利润，同时通过调整报价来中和库存、保持非方向性。

1. 先说结论：这套策略到底是什么流派

如果用交易员语言讲，这个项目的策略不是：

看盘口不平衡后主动追单
看 OFI 后打进去抢 1 tick
预测下一跳方向

它更像：

在中间价两侧同时挂单
谁来打到我，我就先被动成交
然后靠 bid/ask spread、库存回归和反复双边成交赚钱

所以它属于非常标准的 passive market making / inventory-skewed quoting。从 README 和 marketmaker.py 的实现看，这个 agent 的主要目标不是“做对方向”，而是围绕公允价持续报双边，并根据持仓偏离来把报价往不利于继续堆仓的一边推开。

2. 这个项目里的市场环境是什么样

这个项目不是接真实交易所，而是自己模拟了一个市场。README 说得很清楚：交易所端会随机处理 limit orders、market orders，以及可选的 IOC / Post-Only 订单，最后生成一个类似布朗运动的 mid-price。exchange.py 里也能看到，模拟器会：

先在中间价附近铺一组随机 limit order ladder
按指数分布时间间隔打入随机 market order
定期删除旧挂单，控制盘口总量
还会偶发“spike”式跳动，随机把中间价往上或往下推一段，再额外加更多挂单

这说明它研究的不是“真实市场 alpha”，而是一个教学/实验型的被动做市问题：
在一个随机波动、不断有人来吃单的 LOB 环境里，做市商怎么持续挂双边、控制库存、避免越做越偏。

3. 它的策略核心只有 3 步
第一步：先取当前 mid-price

marketmaker.py 里 agent 会先从服务端读取当前 mid-price。这个 mid-price 就是它的定价基准。

第二步：根据当前持仓算一个“无差异价格 / reservation price”

代码里名字叫 indiff_price。核心公式是：

indiff_price=mid_price−price_shift

而

price_shift=(
max_delta
position
	​

)×spread

也就是说：

当你多头持仓为正时，price_shift 为正，indiff_price 会被压低
当你空头持仓为负时，price_shift 为负，indiff_price 会被抬高

这就是做市里非常经典的 inventory skew / reservation price adjustment 思路。

第三步：围绕这个 indifference price 挂双边

代码里直接是：

bid_price = indiff_price - spread/2
ask_price = indiff_price + spread/2

也就是在“库存修正后的公允价”两边，对称挂出 bid 和 ask。

4. 为什么要这样移动报价

这是这个策略的灵魂。

假设你现在已经买了一堆，仓位是正的。
如果你还继续按原 mid-price 对称报双边，会发生什么？

你的 bid 还是太有吸引力，可能继续买进
你的 ask 还不够积极，减仓速度不够快
结果库存越堆越大

这个项目的处理方式是：

一旦持仓偏正，就把整组报价往下移。

这样一来：

你的 bid 更低，不容易继续吃进太多货
你的 ask 也更低，更容易把手里的多仓卖出去

反过来，如果你是空仓偏多，也就是持仓为负，代码会把 indiff_price 往上调，这样：

ask 报得更高，不容易继续加空
bid 也更高，更容易回补空仓

所以这套逻辑不是在赌方向，而是在做：

“库存回归”

5. 它不只是调价格，还会调挂单数量

这点很重要。这个项目不只是“价格偏移”，还做了size skew。

代码里有一段非常直白：

如果 position >= 0，则
bid_volume = max_volume * (1 - position/max_delta)
ask_volume = max_volume
如果 position < 0，则
bid_volume = max_volume
ask_volume = max_volume * (1 + position/max_delta)

这表示：

当你多仓偏多时
买单量会被缩小
卖单量保持最大

意思就是：
少继续买，多卖一点。

当你空仓偏多时
卖单量会被缩小
买单量保持最大

意思就是：
少继续卖，多买一点。

所以这个项目其实同时用了两把库存控制刀：

价格 skew
数量 skew

这比只调价格更稳，因为它同时影响：

你被成交的概率
你一旦被成交时，仓位变化的速度
6. 这个策略的完整交易循环是什么

整个 loop 很简单，但很标准：

读取当前 mid-price
读取自己的持仓 position
根据持仓算 price_shift
得到库存修正后的 indiff_price
计算 bid/ask 价格
根据持仓决定 bid/ask 数量
一次挂出一对 quote
等一小段时间
把这对 quote 删掉
再重新计算并重挂

这说明它是一个 refreshing quotes 的 market maker，而不是永久静态挂单。
README 里的说明和 main.py 的默认参数也能看出，它是定时更新式的；示例参数里 market maker 默认用一个固定 spread、固定 max_volume、固定 max_delta，然后每隔 sleep 秒更新一次。

7. 它赚的到底是什么钱

这套策略的收益来源，不是趋势收益，而是 3 类钱。

第一类：spread capture

最直接的就是 bid/ask 差价。
如果你先在 bid 买到，再在 ask 卖掉，中间就赚 spread。
这个项目的整个设计就是为了不断制造这种双边成交机会。

第二类：库存回归带来的再平衡收益

当仓位偏了以后，代码会把报价往有利于减仓的方向推。
如果市场只是随机波动而不是单边暴走，这种库存回归能帮助它不断回到中性附近。

第三类：被动成交优先于主动成交

这个 agent 下的是双边 quote，类里默认 order_kind 是 post-only；虽然 main.py 示例里传的是 limit，但整体思路明显是被动提供流动性，而不是 taker 主动扫单。

8. 这套策略为什么是“非方向性”的

README 里明确写了它会 neutralize inventory，并 maintain a non-directional strategy。原因就在于：

它不会因为“看涨”就只挂 bid
也不会因为“看跌”就只挂 ask
它一直是双边在场，只是根据库存轻微倾斜价格和数量

这不是 directional alpha，而是 liquidity provision with inventory control。

所以你可以把它理解成：

我不猜你接下来涨还是跌，我只是持续卖“立即成交的便利性”，然后管理自己因为提供流动性而积累出来的库存风险。

1）逻辑非常清楚

它几乎把最经典的 market making 思路压缩到了最小版本：

以 mid 为基准
用 inventory skew 调 reservation price
用 volume skew 控制仓位
定时撤单重挂
2）非常适合教学和验证

README 说它是一个 LOB matching engine + market maker；而交易所端又是可控的随机环境。
这很适合拿来理解：

什么是双边报价
什么是库存风险
为什么做市不能只看 spread，不看 position
为什么做市策略需要不断 refresh quotes
3）它把“库存控制”做得很直白

很多新手 market making 代码只会固定挂两边。
这份代码至少已经把最关键的 inventory skew 做出来了。

10. 它的局限也很明显

这部分我会说得直一点：

这更像一个入门版 market making demo，不是实盘级做市系统。

这是我根据公开代码做的判断。原因有几个。

1）没有显式 alpha 预测

从 README 和 marketmaker.py 的核心逻辑看，它没有用：

order imbalance
OFI

短期波动预测

所以它不是“带 alpha 的 maker”，而是更纯粹的被动赚 spread。这是根据公开代码作出的直接判断。

2）spread 是固定的

main.py 示例里 spread 是固定值。
这意味着它没有随着：

波动变大
市场更薄

而动态扩宽或收窄报价。

3）更新是定时的，不是事件驱动的

它是挂单 -> sleep -> 撤单 -> 重挂。
这对教学没问题，但离真实高频 market making 的“事件驱动、逐笔响应”还差很远。

4）市场环境是随机仿真，不是真实微结构

交易所端是随机 market orders、随机 limit ladder、随机 spikes，再加一个布朗式 mid-price。
这说明它适合学机制，不适合直接拿来证明“这套策略在真实股票市场一定赚钱”。

1) Pure Market Making，PMM

这是 Hummingbot 最基础、最经典的策略。核心就是：在一个现货交易对上，同时挂 bid 和 ask，价格相对 mid-price 偏离 bid_spread 和 ask_spread，然后每隔 order_refresh_time 秒撤掉旧单、按新价格重新挂单。官方文档还写明它是按 tick 运转的，默认理解就是周期性刷新报价。

盘口两边都挂单；
别人来吃你的单；
你赚 spread；
再通过 inventory skew、refresh 等机制，避免仓位越来越歪。

这类策略适合：

你不想强烈押方向；

单边行情把你库存打歪；
你挂单后价格快速穿过去，来不及撤；

所以 PMM 本质上是 maker 策略，不是 taker。

举个直观理解：
如果你库存偏多，它会把 reservation price 往下压，或者让 ask 更积极、bid 更保守，目的是把库存卸掉；如果波动率变大，它会倾向把 spread 拉宽，减少被不利成交的风险。

所以它仍然是 maker 做市，但比 PMM 更强调：

库存控制；
风险厌恶；
波动率自适应。

适合的人：

想做系统化做市；
不满足于固定 spread；
愿意理解库存风险和波动率之间的关系。
不适合的人：
只想靠极简参数快速跑；
不想碰模型参数。
3) Perpetual Market Making

这个是把 PMM 的思想搬到永续合约上。官方文档写得很清楚：它是在单个 perp 订单簿上持续挂限价买卖单，等待别人来成交；和现货不同的是，永续市场里你不是简单交换资产，而是在开仓/平仓/持仓。

本质上，它还是 maker spread capture，只是场景从 spot 变成 perp。
好处是：

perp 流动性常常更强；
可用杠杆；
更容易做双向仓位管理。

但新增风险也很明显：

funding rate；
杠杆放大亏损；
强平风险；
perp 市场常有更激烈的短期波动。

所以这类策略适合做：

合约市场做市；
想在 perp 上赚 spread。
不适合做：
无经验高杠杆裸跑。
二、跨市场做市 / 做市+对冲类
4) Cross-Exchange Market Making，XEMM

这是 Hummingbot 很核心的一类策略。官方把它定义成：在 maker 交易所挂单做市，一旦成交，就去另一个 taker 交易所立刻做反向对冲。它追求的是 maker 端赚到的 spread，要比 taker 端吃掉的成本更大，中间至少留下 min_profitability。

它的结构非常清晰，官方文档直接拆成两大块：

Order creation and adjustment：周期性创建和调整 maker 端挂单；
Hedging order fills：maker 端一成交，就在 taker 端做对冲。

它和普通 PMM 最大的不同在于：
PMM 成交后，你直接暴露在单边库存里；
XEMM 则是在另一个更深、更紧的市场里把风险对冲掉。

官方还写了它的报价不是瞎挂，而是会参考：

taker 侧当前市价；
maker 侧当前订单簿深度；
top_depth_tolerance；
min_profitability。
如果挂着的 maker 单已经不再是“最优可盈利价格”，它会取消重挂。

这个策略的本质是：

在流动性差一些、spread 大一些的市场做 maker；
在流动性好一些、spread 小一些的市场做 taker；
把“深市场的流动性”搬运到“浅市场”。

它非常适合你理解成 maker/taker 混合策略。
maker 端赚钱，taker 端止风险。
主要风险：

对冲腿没打出去；
taker 端滑点大；
两边 API/延迟不同步；
DEX 一侧还有 gas、可靠性问题。官方也专门提醒了 DEX 侧会更不稳定，某些操作可能要重复执行。
5) Cross-Exchange Mining

这个名字看着像 XEMM 的兄弟，实际上也确实很像。官方文档说，它同样是在 maker 端挂比 taker 端更宽的 spread，maker 一旦成交，就到 taker 端做平衡交易，目标还是保住 min_profitability。但它比 XEMM 更强调一件事：持续跟踪两边 base asset 数量并做再平衡，尽量维持固定的 order_amount 基础库存，从而减少因为市场涨跌导致的 base/quote 暴露风险。

你可以把它理解为：

XEMM 更偏“跨所做市 + 立刻对冲”；
Cross-Exchange Mining 更偏“跨所做市 + 库存守恒 + 持续再平衡”。

适合场景：

两个现货 CEX 之间做搬运式做市；
你很在意长期库存不要越跑越偏。

它主要还是 maker 为主，taker 为辅。

三、纯套利类
6) AMM Arbitrage

这个策略是典型的 跨市场价差套利。官方写得很直接：它监控一个 AMM DEX 上的交易对价格和另一个 AMM/CEX/DEX 上的同交易对价格，只要套利空间在扣掉 gas 和交易手续费以后仍然高于 min_profitability，就同时在两个市场执行对冲的买卖单，锁定价差。

它的核心不是挂单等人吃，而是：

发现两个市场同一资产价格不一致；
一边买、一边卖；
套走中间差价。

所以它本质上偏 taker/taker arbitrage。

最关键的点有三个：

价差是否真实存在；
扣完 gas 和手续费是否还赚钱；
执行时两个腿能不能都打到。

主要风险：

DEX 交易确认慢；
gas 飙升；
链上价格在你打单前已经回归；
CEX/DEX 两边资产预先分配不合理。

如果你做的是链上 + 中心化市场联动，这个策略很有代表性。

7) Spot Perpetual Arbitrage

这个是 现货-永续基差套利。官方描述是：先看 spot 和 perp 的价格 spread，当 spread 高于 min_opening_arbitrage_pct 时，就开一组对冲仓位——在一边买/卖现货，同时在另一边做相反方向的永续多/空；等到两边价差收敛到 min_closing_arbitrage_pct 以下，再把两边一起平掉。

这个策略的交易逻辑非常标准：

开仓条件：价差偏离够大；
持有过程：等价差回归；
平仓条件：价差收敛到目标。

它赚的不是盘口 spread，而是 basis convergence。
关键参数也很明确：

min_opening_arbitrage_pct
min_closing_arbitrage_pct
perpetual_leverage
两边 slippage buffer
cycle delay。

这类策略很像传统统计套利/基差套利，而不是做市。
主要风险：

价差不回归；
funding rate 吞利润；
perp 仓位管理出问题；
杠杆把本来稳健的套利变成危险交易。
四、风险管理 / 辅助型策略
8) Hedge

Hedge 不是拿来单独“赚钱”的主策略，它更像 风险覆盖层。官方写得很清楚：这个策略允许你对另一个做市策略进行对冲，可以在 perp 交易所或另一个 spot 交易所开相反仓位，hedge_ratio 这类参数用来控制对冲多少；而且官方明确说，用户应当把这个策略和另一个 market making strategy 一起运行。

所以 Hedge 的定位是：

不是主动找 alpha；
而是给已有策略降库存风险。

比如你用 PMM 在现货里越做越多头，那 Hedge 可以在 perp 里做空一部分，把净暴露压下来。

这在你做多策略组合时很有用，因为很多做市问题最后不是“策略没成交”，而是“成交后库存太偏”。

五、激励型流动性提供
9) Liquidity Mining

这个是 Hummingbot 很有代表性的概念。官方 V1 文档说，这个策略可以在 同一个实例里、同一个交易所上、跨多个交易对做市；而且有一个关键特征是 volatility-spread adjustment，也就是根据市场波动动态调 spread。参数里你能看到 markets、token、inventory_skew、volatility_interval、avg_volatility_period、volatility_to_spread_multiplier、max_spread 这些东西。

它和普通 PMM 的差异在于：

PMM 更像“单对、单策略”；
Liquidity Mining 更像“多对、多资产配置、顺便争取奖励”。

Hummingbot 当年还专门围绕这个提出过“流动性挖矿”模式：让普通用户也能通过跑做市机器人为某些币对提供流动性，并拿到额外奖励。官方白皮书摘要提到，这个模型围绕订单簿快照、spread 密度函数、预算分配机制来设计奖励；早期材料里甚至给出过 10%–50% 年化这类模拟回报讨论，但那属于历史材料，不应该被当成今天的确定收益承诺。

这个策略适合：

你想同时跑多个交易对；
你愿意把“做市 PnL + 平台奖励”一起算。

主要风险：

奖励下降后，裸做市可能不赚钱；
多对同时跑，管理复杂度高；
趋势市里 inventory 风险依然在。
六、Hummingbot 现在真正该怎么理解：V2 框架

官方现在已经把重心放到 Strategy V2。V2 的思路不是“给你几个写死的完整策略”，而是：

Controllers 负责决定什么时候交易；
Executors 负责怎么把交易执行出去。

官方对 Executors 的定义也很明确：它们是自管理组件，负责下单、刷新、撤单、停止；而 Controllers 则利用 MarketDataProvider 提供的 Candles、Orderbook、Trades 数据来产生条件。现成的 executor 类型包括 Position、DCA、Grid、Arbitrage、TWAP、XEMM、LP。

这意味着什么？

如果你只是想快速上线一个做市/套利 bot，看 V1 思路够了。
如果你想做更复杂、更接近你自己研究框架的策略，应该看 V2。

比如官方现在已经有：

XEMM Executor：做跨所做市/价差捕捉；
Arbitrage Executor：评估成本后做套利；
LP Executor：做 CLMM DEX 的自动 LP；
还有样例脚本分类，包括 data/screening、execution/PMM、V2 orchestration。

而且 2025–2026 的 release notes 也持续在强调 V2 的统一化和 LP/套利等能力扩展，说明这不是旧概念，而是当前官方主线。

八、它和你关心的“微观高频策略”是什么关系

如果你说的“微观高频”是你前面一直在研究的那种：

queue imbalance
OFI
tape

快进快出

那我会直接说：

Hummingbot 内建策略并不是天然为这个方向设计的。
它更像一个 加密市场的做市/套利自动化框架，而不是一个 L2/L3 微观结构高频研究平台。这个判断主要来自官方策略的核心参数与执行框架：它们主要围绕 spread、inventory、hedge、min_profitability、rebalancing、volatility adjustment、order refresh 来设计。

但它也不是完全不能做“偏微观”的东西。
因为官方 V2 架构说明，Controllers 可以利用 Orderbook 和 Trades 数据，再驱动 Executors 去执行。也就是说，你可以把 Hummingbot 当执行框架，把你自己的微观信号塞进 Controller 里。

所以更准确的说法是：

拿现成策略直接跑：更适合做市/套利；
拿 V2 自己改：可以向微观短线靠近；
但如果你要真做毫秒级盘口高频：Hummingbot 不是最理想底座。

hftbacktest策略 思路全面讲解
Thought for 2m 0s

hftbacktest 先要纠正一个点：它不是一套单独的策略，而是一个偏微观结构/HFT 的高精度回放式回测框架。它的重点是把 行情延迟、下单延迟、队列位置、L2/L3 订单簿重建、成交模拟 做得更接近真实交易，用来研究高频做市和微观 alpha。官方文档里展示的策略，绝大多数也都围绕这条主线展开。

你可以把 hftbacktest 里的“策略思路”理解成一条统一母线：

先估计短期公平价格 / 下一小段时间的价格压力 → 再决定 bid/ask 报价位置 → 再根据库存风险、队列位置、延迟、极端行情保护去动态撤改单。
所以它本质上是“定价 + 做市执行”系统，不是那种纯技术指标择时系统。官方教程里从简单高频网格、GLFT 做市，到 OBI、Basis、APT、多市场组合、Large Tick 队列做市，基本都在这个框架里演化。

1. hftbacktest 里最核心的策略类型
1）高频网格做市（High-Frequency Grid Trading）

这是最基础的一类。思路不是预测大趋势，而是在中间价附近，按固定 tick 间隔两边挂一排单，赚买卖价差 + rebate。官方教程里有 plain grid，也有带 skew 的版本。

它的核心逻辑很简单：

以当前 mid 或某个 reservation price 为中心
上下各挂多档限价单
被动成交后，在另一侧继续补挂
通过控制网格间距、层数、每档手数、最大库存，来平衡收益和风险

为什么它适合放在 hftbacktest 里研究？因为这类策略对成交模拟极度敏感。你到底能不能在 best bid / best ask 排到前面，延迟大一点会不会排不到，撤单有没有来得及，这些都会直接决定策略从赚钱变成亏钱。hftbacktest 恰好就是在这些地方做精细建模。

你可以把它理解成：

不是靠“方向预测赚大波段”，而是靠“不断低买高卖、吃点差和返佣”积小胜。

2）带库存倾斜的网格/做市（Grid with Skewing）

这比纯网格更实战。官方高频网格教程里专门有 skewing 变体。

核心思想是：

当你已经买了很多货，说明库存偏多
这时继续积极挂 bid 会危险，所以要：
把 bid 往下收
把 ask 往更容易成交的位置推
当你空仓或偏空时，反过来做

也就是把库存风险编码进报价。
这类策略的目标不是“永远对称挂单”，而是：

仓位越偏，报价越偏；让市场帮你把仓位拉回中性。

这是后面 GLFT、Avellaneda 类做市的基础思想。

3）GLFT / Avellaneda 风格做市

官方教程里有 Guéant–Lehalle–Fernandez-Tapia Market Making Model，并明确提到库存风险、skew、fair value pricing 这些关键点。文档还指出，在这个模型里公平价格默认等于 mid，但实际中需要叠加 micro-price 或相关资产 fair value 预测来增强策略。

这类策略的核心不是“机械挂网格”，而是：

先算reservation price（保留价格）
常见形式：reservation price = fair value - inventory_penalty
再算optimal spread
波动大、成交不确定时，spread 要变宽
波动小、成交机会大时，spread 可缩窄
然后围绕 reservation price 去挂 bid/ask

所以 GLFT/Avellaneda 类的本质是：

“价格中心 + 风险惩罚 + 最优点差” 三件套。

它比朴素网格高级的地方，在于：
不是死板地每档相隔 1 tick，而是根据市场环境和库存风险，自适应调整。

4）订单簿不平衡驱动的 Alpha 做市（Order Book Imbalance）

这是 hftbacktest 很重要的一支。官方有独立教程 Market Making with Alpha - Order Book Imbalance，而在后续 Pricing Framework 里又再次验证了 standardized OBI 的有效性。

思路是：

看盘口买量和卖量谁更强
如果买盘明显更厚，短期公平价往上偏的概率更大
如果卖盘更厚，短期公平价往下偏的概率更大

于是你不再简单围绕 mid 报价，而是围绕一个向强势一侧偏移的 fair value 报价。

这类策略的实际含义是：

买盘强：
你的 bid 可以更积极一点，ask 可以更高一点
卖盘强：
你的 ask 更积极，bid 更保守

也就是说：

OBI 不是直接拿来做“突破追涨杀跌”，而是拿来修正做市中心价。

这点很关键。
在 hftbacktest 里，大多数 alpha 信号都不是单独下市价追单，而是变成报价系统的一部分。

5）Basis 做市（现货-期货基差驱动）

官方有 Market Making with Alpha - Basis，核心就是先用 L1 book ticker 预计算 spot 和 futures 的 basis，再拿 basis 去修正做市的定价。

这类策略的逻辑是：

若期货相对现货明显偏贵
futures 的 fair value 可能要下修
若期货相对现货明显偏便宜
fair value 可能要上修

所以本质上是：

用跨市场相对价格偏离，替代“只看本市场盘口”的单市场定价。

这类策略比 OBI 更“跨市场”：

OBI 看的是本盘口内部供需
Basis 看的是相关市场之间的相对错位

适合：

期现联动强
现货有价格发现能力
期货存在短时偏离
6）APT 做市（多因子 / 多市场定价）

官方把 Market Making with Alpha - APT 作为 Basis 的延伸，直接说这是基于 Arbitrage Pricing Theory 的 market making。页面里还拆成了 Core Market Drivers、Broader Market Influence、Microstructure Signals & Alpha Factors、Combined Pricing Model 等部分。

这类策略可以理解成：

把多个 alpha 因子线性/回归式组合，形成更强的 fair value 预测。

官方的 Pricing Framework 还展示了几类因子来源：

现货 return 信号
主市场/主驱动市场 return
VAMP / Effective VAMP
standardized OBI
多信号加权组合

并且文档明确提醒：多因子调权虽然可能提高结果，但会明显增加过拟合风险。

所以 APT 在 hftbacktest 里的真实含义，不是学术上的 CAPM 替代品，而是：

“把多个微观/跨市场信号融合成一个短周期 fair value 引擎”。

这是我觉得它最值得学的地方。
因为它已经不是单因子做市，而是在往“微观结构定价框架”升级。

7）Large Tick 资产的队列做市（Queue-Based MM）

官方有一整篇 Queue-Based Market Making in Large Tick Size Assets。文档直接说：在大 tick 资产里，队列位置特别重要，因为价格运动更受 tick 尺寸约束。教程里还给出 book pressure、trade impulse、pure queue-based model。

这类策略的核心不是“谁方向更对”，而是：

谁更容易先成交，且成交后还能不吃亏。

大 tick 市场里，常见现象是：

spread 经常卡在 1 tick
价格不太容易连续跳
谁排在 best 上更前，谁就更容易拿到那一点点 edge

所以这类策略非常重视：

当前 best bid / ask 队列厚度
自己在队列里的位置
对手盘主动成交速度
撤单/改单导致的队列前移概率

文档里的示例还把 book_pressure 类似 micro-price 的想法引进来，并用 reservation price + skew 的方式做报价。

这类策略特别接近你之前反复关心的那种“微观 edge”问题：

不是看大趋势，而是看未来几秒、甚至几百毫秒，哪边更容易先动，自己排进去是否更有成交优势。

8）多市场/多品种组合做市（Making Multiple Markets）

官方教程明确说：通过对多个资产做市并构造 book，可以获得更好的风险调整后收益，原因是分散化。还强调：哪怕单个策略 Sharpe 不高，把多个低相关资产或策略组合后，组合 Sharpe 也可能更高。

这不是“单个标的怎么做”，而是“做市组合怎么做”。

核心思想：

单个币对的 edge 很小也很不稳定
但如果 10 个、50 个、100 个资产同时跑
只要收益相关性没那么高
组合后的曲线会更平滑

这也是很多真 HFT 团队的常识：

单策略 edge 不一定惊艳，真正厉害的是规模化复制 + 分散。

2. hftbacktest 这些策略的共同骨架

无论上面哪一种，实际都能拆成 6 步：

第一步：定义 fair value

可能来自：

mid price
micro-price / book pressure
OBI
basis
现货 return
多因子线性组合

官方 Pricing Framework 甚至明确把“主市场价格驱动、跨资产影响、微观结构 alpha、多因子组合”都放进了统一框架里。

第二步：算 reservation price

公平价不是最终报价。
还要减去库存惩罚项，得到真正想报价的中心。

也就是：

多头太多 → reservation price 下修
空头太多 → reservation price 上修
第三步：算 spread / half spread

依据：

波动
成交强度
tick size
fee / rebate
latency 风险
第四步：生成 bid / ask 网格

围绕 reservation price 放 1 层或多层报价。

第五步：看队列与延迟

这一步是 hftbacktest 最有价值的地方。
官方强调它的设计目标就是把 feed latency、order latency、queue position 放进回放回测。

第六步：撤单、改价、风控退出

行情反向、延迟飙升、basis 异常、last 和 mark/spot 偏离过大时，收窄仓位甚至退出市场。官方也有专门的 Price Protection 教程，建议在极端行情下用更保守的定价、异常时退出、并监控 feed/order latency 的尖峰。

3. hftbacktest 为什么和很多“量化框架”不一样

它的卖点不在“策略库多”，而在“执行仿真更贴近真实”。

官方首页直接写了几个关键点：

tick-by-tick simulation
L2 / L3 订单簿重建
latency modeling
queue position fill simulation
multi-asset / multi-exchange
还能把同一算法代码用于 live bot 原型，但目前 live 侧主要是 Rust，面向 Binance Futures 和 Bybit。

这意味着它特别适合研究这类问题：

best 上挂单到底能不能排到
延迟增加 20ms，成交率掉多少
订单回报变慢时，撤单还来不来得及
large tick 资产里，队列比方向预测更重要吗
多因子 fair value 能否覆盖 maker rebate 后的 edge

这也是为什么它不像 Hummingbot 那样“偏生产 bot”，而更像：

偏研究/回测基础设施。

4. 订单成交模拟，是它最值得学的地方

官方 Order Fill 文档说得很清楚：想把回测做准，队列位置很重要，尤其在没有 MBO、只有 MBP 数据时，就必须用模型去猜。文档提供了保守的 RiskAverseQueueModel，也提供了概率型 ProbQueueModel 及其多个变体。

你可以这么理解：

RiskAverseQueueModel

最保守。
只有同价位真实成交发生时，你的队列才前进。
适合不想高估成交率时用。

ProbQueueModel

更现实一点。
如果盘口量减少，不一定全是你后面的人撤掉，也可能前面有人撤，所以你的队列位置可以按概率前移。官方说明还引用了 queue position estimation 的经典资料。

这对做市策略影响巨大。
很多“纸面赚钱”的 maker 策略，问题根本不在信号，而在：

你以为你会成交，实际上你排不到。

5. 延迟，在 hftbacktest 里不是附属品，而是核心变量

官方首页和教程都反复强调 latency。框架的重点之一就是同时考虑 feed latency 和 order latency。

为什么这点重要？

因为高频做市不是“信号对了就一定赚钱”，而是：

你看到的 best bid/ask 是不是已经过时
你的改单到交易所时，市场是不是已经变了
你撤单时，对手是不是已经先打到你

官方还有 Accelerated Backtesting，明确说它为了提速，会忽略部分队列填单和 order-response latency，所以速度更快，但准确性会下降，尤其当 queue position 很关键时会更明显。

这说明 hftbacktest 自己也很诚实：

快速研究可以牺牲一些细节
真正验证 edge，还是要回到高精度模式
6. 用一句话概括每种策略

如果你想快速抓重点，可以这么记：

高频网格：围绕中间价双边挂单，赚点差和返佣。
Skew 网格：库存偏了就把报价往有利于减仓的方向倾斜。
GLFT/Avellaneda：用 reservation price + optimal spread 做自适应做市。
OBI 做市：用盘口不平衡修正短期公平价。
Basis 做市：用现货-期货基差修正期货 fair value。
APT 做市：把多个微观/跨市场因子组合成 fair value 引擎。
Queue-based 做市：大 tick 资产里，排队优势本身就是核心 edge。
多市场组合做市：单资产 edge 小，但多资产组合后风险收益比更好。
7. 它更偏 maker，还是 taker？

就官方教程风格看，hftbacktest 的示例明显更偏 maker / quote-driven，也就是：

挂限价单
研究队列
研究成交概率
研究库存和撤改单

而不是那种：

信号一来立刻市价吃单
持仓几百毫秒后马上止盈止损

当然，框架本身不是不能做 taker，只是它最强的价值，确实在 maker 微观执行仿真。这从官方对 queue position、order fill、latency 的强调就能看出来。

所以如果你问：

hftbacktest 最适合研究什么？

我的答案是：

最适合研究“微观结构做市类 edge”，尤其是那些盈利很薄、对排队和延迟极其敏感的策略。

8. 对你这种“微观高频策略”研究，最值得抄的框架

结合你一直在研究的“盘口不平衡 / microprice / OFI / queue edge”方向，我觉得从 hftbacktest 最值得抄走的，不是某一段策略代码，而是这套研究范式：

研究范式 A：先做 fair value，不急着做入场规则

先不要问“什么时候买”。
先问：

当前 fair value 是多少
它相对 mid 偏上还是偏下
这个偏移有多大
这个偏移能不能覆盖手续费、点差、滑点、延迟成本
研究范式 B：把信号放进报价，不一定直接追单

很多信号最好的用途不是直接买入，而是：

调整 quote center
调整 spread
调整 skew
决定是否收缩仓位
研究范式 C：把执行问题当成 alpha 的一部分

不是“信号”和“执行”分开。
而是：

能否排到、何时排到、能否及时撤掉，本身就是 edge 的组成部分。

研究范式 D：先验证真实可成交，再谈参数优化

官方首页明确强调：在谈 overfitting 之前，先确保回测能尽量贴近真实执行，最好能和 live 结果靠近。

这个思路非常对。

9. 我对 hftbacktest 的一句总评

如果 Hummingbot 更像：

“把做市/套利机器人跑起来”

那 hftbacktest 更像：

“把微观结构 edge 研究透，再确认这 edge 在真实成交机制下还剩多少”。

所以它真正厉害的不是“策略花样多”，而是：

它把高频策略最容易被忽略、但最致命的部分——队列、延迟、成交模拟——摆到了台面中央。

《hftbacktest 全部策略总表：策略名 / 核心信号 / 做市逻辑 / 更偏 maker 还是 taker / 优势 / 风险 / 适合什么市场》

tiagosiebler/TriangularArbitrage策略 思路全面讲解
Thought for 1m 18s

这个仓库的核心，不是“跨交易所搬砖”，而是单交易所内的三角套利机会检测器。README 写得很直接：它在同一个交易所里通过 websocket 监控多个币种，计算所有可能的三角路径 ab -> bc -> ca，再扣掉手续费后把机会按收益率排序，并且可以把历史机会存进数据库做后续分析；配置里还写明目前只支持 binance。

一句话概括它的策略思路：

拿一个起点币，当作本金；沿着两条中间桥梁币和一个第三币兜一圈，如果最后回到起点时数量大于 1，再扣掉手续费后仍然大于阈值，就认为出现了三角套利机会。

1. 这套策略本质上在做什么

三角套利的基本形式是：

先拿起点币 A
换成桥梁币 B
再换成第三个币 C
最后把 C 换回 A

如果三步乘起来的汇率乘积大于 1，就意味着账面上有正收益。这个仓库默认起点是 BTC，默认桥梁币是 BNB, ETH, USDT，这些都写在 conf.ini 和启动参数里。程序界面标题里也会直接显示 Top Potential Arbitrage Triplets, via: BNB,ETH,USDT。

所以它不是那种“预测涨跌”的策略，而是：

找同一交易所内，不同交易对之间的瞬时定价不一致。

你可以把它理解成一种图论式汇率闭环检查：
只要 A→B→C→A 这条闭环在当前最优买卖价下跑出来是盈利的，它就会把这条路径挑出来。

2. 它是怎么找三角路径的

代码里把路径拆成：

start: 起点币，默认 BTC
paths: 中间桥梁币，默认 BNB, ETH, USDT。

然后它会从 Binance 的 all market tickers websocket 流里拿到全市场 ticker 数据，再把这些 ticker 同时做成：

原始数组
以交易对 symbol 为 key 的对象
按某个币分组后的子市场集合。

接着它的思路是：

先固定起点，比如 BTC
再固定一个桥梁币，比如 ETH
去市场里找所有既和 BTC 有关系、又和 ETH 有关系的第三币 X
这样就能构造出类似：
BTC -> ETH -> X -> BTC
或 BTC -> BNB -> X -> BTC
或 BTC -> USDT -> X -> BTC 的闭环。

所以它的“广度”来自于：

不是只写死一条三角路径，而是用起点币 + 桥梁币集合，动态枚举所有能闭环的第三币。

3. 它怎么判断每一步到底是买还是卖

这是这个仓库最关键的地方之一。CurrencyCore.getCurrencyFromStream() 里面实际上做了一个方向判断：
如果找到的是 toCur + fromCur 这种符号，代码把这一步视为买入，用的是 ask 价；如果找到的是 fromCur + toCur 这种符号，代码把这一步视为卖出，用的是 1 / bid。同时它还给每一步打上 BUY 或 SELL 标记。

这背后的交易含义非常重要：

买某个币时，要吃 ask
卖某个币时，要打 bid
如果交易对方向和你想走的方向反过来，就要做倒数变换

这说明它在算套利时，不是拿中间价乱乘，而是尽量按“真实可交易方向”的最优一档报价去算。

然后 getArbitageRate() 会把三步的 rate 直接相乘，得到整条三角路径的总乘积。也就是：

总收益倍数 = 第一步成交汇率 × 第二步成交汇率 × 第三步成交汇率。

4. 这个策略最核心的“信号”其实只有一个

它不是多因子，不看 K 线，不看均线，不看盘口不平衡。
它的唯一核心信号就是：

三步汇率乘积是否大于 1，并且扣完手续费后是否仍然有剩余。

README 里展示的输出也很明确，表头包括：

Rate
Fees BnB
(Rate - BnB Fee)
Fees Normal
(Rate - Fee)。

这说明它在展示时会同时给出：

不同手续费模式下的成本
扣费后的净机会大小

所以这类策略的本质不是“找趋势”，而是：

找价格闭环错误。

5. 为什么它默认用 BTC + BNB/ETH/USDT 这套结构

因为在老 Binance 现货市场里，这几个币长期承担了大量“计价桥梁”的角色。
默认配置里写的是：

binanceStartingPoint=BTC
binanceColumns=BNB,ETH,USDT。

所以它的策略思路其实是：

先从一个主资产出发
再通过几个最常见的桥梁币去扩展
把所有能闭环的币都找出来

这种方式的优点是：

枚举空间够大
但又不会大到完全失控
而且更容易找到真实存在流动性的路径。

也就是说，这不是随便全市场暴力三元组枚举，而是带有桥梁币优先级的三角搜索。

6. 它为什么还要做 pair ranking 和 hit threshold

这部分很容易被忽略，但其实很实战。

PairRanker

PairRanker 会把每次出现的候选路径都记录成带时间戳的对象，然后把超出 pairTimer 时间窗口的数据清掉。默认 pairTimer=900000，也就是 15 分钟。它还会检查前几个候选路径的平均 rate 是否高于 minimalProfit，默认配置里这个值写的是 1.02。

这表示它不只是看“这一瞬间有没有利润”，还在看：

这个路径是不是在最近一段时间里也经常偏高。

换句话说，它想过滤掉那种：

只闪一下
噪声很大
难以执行

的机会。

Trading queue

启动参数里还有：

paperOnly: true
minQueuePercentageThreshold: 3
minHitsThreshold: 5。

TradingCore 里会先把大于阈值的候选路径放进队列，再统计它出现了多少次；只有当同一路径多次命中，并且再次实时验证时仍满足阈值，才会触发 newTradeQueued 事件。

这套设计背后的策略思想非常清楚：

先不急着交易，先确认这个套利机会不是一瞬间的假象，而是反复出现、具有一定持续性。

这其实是在做一种“机会去噪”。

7. 但这个仓库有一个很重要的现实点：它更像“检测器”，不是完整自动套利系统

虽然名字叫 TriangularArbitrage，而且代码里有 TradingCore，但当前仓库默认就是 paperOnly: true，而且 initiateTrade() 还是空的，processQueue() 里也只写了“Plan”占位，没有完整下单流程。 README 更强调的是监控、计算、排序、存库分析，而不是完整执行。

所以更准确地说，这个项目是：

三角套利机会扫描器 + 统计分析骨架

而不是：

生产级自动下单套利引擎。

这点很关键。
因为很多人看这种 repo 会误以为“能直接赚钱”，但实际上它离真实可执行还有一大段距离。

8. 这个策略的真正优势在哪

它的优点很鲜明：

第一，逻辑简单直接

不需要预测市场涨跌，只需要找闭环报价错误。
只要三步回来比出发多，就有理论套利空间。

第二，实时性强

它走的是 websocket，全市场 ticker 持续更新，README 和代码注释都表明它是实时流式处理；配置里对 raw ticks 的说明还写了“Usually a tick every 1s”。

第三，天然适合做统计分析

README 说可以把历史机会存到 DB 里做 aggregation & analytics；配置里也有 useMongo=true、ticks_arbitrage、ticks_raw、pair_ranking 这些表名。

第四，路径筛选思路很实战

它不是见到一点价差就冲，而是加了：

最小利润门槛
时间窗口平均值
hits 次数确认
queue 再验证。
9. 这个策略的主要缺陷，也很明显
1）它只看 ticker 顶层报价，不看深度

代码里用的是 all market tickers 流，并且计算时只用了最优 bid/ask 和它们的倒数，没有用 order book depth 做逐档吃单模拟。

这意味着一个很现实的问题：

账面套利 ≠ 可执行套利。

因为你真正下三笔市价单时，可能：

第一笔吃掉一档后价格就变了
第二笔深度不够
第三笔回来时机会已经消失了

所以这个项目更适合发现“表面机会”，不适合直接当作“真实成交收益”的精确估计器。这个结论是基于代码读取方式做的推断。

2）只做单交易所内套利

README 和配置都写得很清楚，目前只支持 Binance。它不是跨交易所搬砖，也不处理提币/转账/对手方风险那套问题。

3）执行层基本没完成

有队列、有 hits、有 trade placeholder，但没有完整自动下单闭环。

4）它属于“老 Binance 生态风格”

默认起点是 BTC，桥梁币是 BNB/ETH/USDT，这更像老现货市场结构；今天的主路径和费用结构未必仍然最优。这个是我基于当前仓库默认配置做的策略层判断。

10. 你可以怎么理解它的完整流程

把它翻成人话，就是下面这条链：

全市场 ticker 流入 → 按币种组织交易对 → 动态找出所有 起点币-桥梁币-第三币-起点币 闭环 → 按买 ask / 卖 bid 的真实方向算三步乘积 → 扣手续费 → 排序显示 → 记录历史表现 → 对高收益且重复出现的路径做额外筛选。

所以它的核心不是复杂算法，而是三件事：

路径生成
真实方向汇率换算
阈值与稳定性过滤
11. 我对这个仓库的总评

如果你问我它的策略本质，我会这样总结：

它是一个“单交易所三角套利机会雷达”，重点在发现和排名，不在完整执行。

它最值得学的不是“能直接跑起来赚钱”，而是下面这套思路：

用 websocket 持续扫描全市场
用交易对方向自动推导买卖路径
用三步乘积检测闭环定价错误
用手续费和时间稳定性去过滤虚假机会
再把数据落库，做长期统计和路径排名。

如果你是拿它做研究或二开，它是个不错的起点。
如果你想直接拿它上实盘，那还差至少这些层：

深度与滑点建模
三腿原子性/顺序执行设计
失败补救逻辑
真实手续费与持仓校验
风控与限额
更低延迟的数据与执行通道。

这些是我基于代码结构做出的工程判断。

5）这套逻辑为什么叫“均值回归”

因为它在赌一个最朴素的统计假设：

当短期价格偏离最近样本均值太多时，这种偏离是“暂时的”，后面更可能回到均值附近。

z-score 本质上就是在量化“偏离均值多少个标准差”。
所以当它高于某个 overbought_threshold，策略把它解释成“涨过头了”；低于 oversold_threshold，就解释成“跌过头了”。(github.com

换句话说，这个模型是在做：

“极端偏离 → 反向交易 → 等均值回归。”

如果偏得太低，就反手做多

所以它不是在预测“趋势继续”，而是在押注：

例如：

RSI < 30 → 短期超卖
RSI > 70 → 短期超买

所以 RSI 在这类系统里通常承担的是：

做多：价格在 MA 下方，且 RSI 进入超卖
做空：价格在 MA 上方，且 RSI 进入超买

本质就是：

“偏离均值 + 动量极端 = 反向进场。”

这是标题里“RSI + MA + Mean Reversion”最合理的策略解释。

先用流动性过滤市场，再用 ADF 在“趋势 / 均值回归”两种模式间切换，然后用短长均线关系 + 斜率阈值来决定开仓，最后配合固定止盈止损和信号反转平仓。 这套逻辑主要写在 LiqMAStrategy 里。

1）这套系统的整体架构

这套程序的结构很像一个简化版事件驱动交易系统：

StreamingForexPrices 负责从 OANDA 流式拿 tick 价格，收到后生成 TickEvent2
LiqForex 负责定时拉历史 K 线，计算流动性指标，生成 LiqEvent
LiqMAStrategy 收到 tick 后做信号判断
Execution 收到 ORDER 或 CLOSE 事件后去下单或平仓
trade() 是总事件循环，统一消费队列里的事件并路由处理。

这意味着作者想做的，不只是“算一个指标然后下单”，而是搭一个能持续跑的自动交易骨架。这个思路是对的：行情、信号、执行分离，后面更容易扩展。

2）它支持哪些交易品种

主程序里不是全市场通用扫描，而是手动指定交易对。代码里直接写了 pairs = "EUR_USD"，同时也支持 USD_CNH 和 AUD_USD，并按品种加载各自参数文件。也就是说，这不是“一个统一自适应模型”，而是 每个品种单独调参数 的做法。

这点很关键，因为它说明作者的交易思想不是机器学习式泛化，而是更传统的：

“不同外汇对有不同微观行为，所以阈值、窗口、止盈止损都要单独调。” 这是很多实盘系统常见的路线。

3）第一层过滤：流动性过滤

这套策略不是看到价格动就进。它先看流动性。

LiqForex 会去请求 /v1/candles，参数里用的是 count=20、granularity='M15'，再基于历史 OHLCV 算一个 liq 指标，然后把它放进 LiqEvent。主程序里 period=600，也就是大约每 10 分钟刷新一次流动性指标。

策略在真正考虑开仓前，会先检查：

pd["ticks"] > max(long windows)+1 and liqIndex > self.idxU

也就是说，样本长度够了，而且当前流动性高于阈值，才允许开始交易。对 EUR/USD 的参数文件来说，这个阈值 IDXU = 220。

这层设计背后的思路非常实战：

在流动性差的时候，均线、斜率、均值回归这些信号很容易被点差和噪声毁掉；先筛掉“难做”的时段，再谈方向。 这一步其实比单纯堆指标更重要。这个判断是我根据代码结构做的归纳。

4）第二层：用 ADF 在“趋势”和“均值回归”之间切换

这是这份代码最有意思的地方。

在没有持仓、且满足流动性条件后，策略会先计算：

critAdf = getADF(collection).priceADF(200,1)

然后直接判断：

critAdf > adf_thres → pd["stra"] = "reversion"
否则 → pd["stra"] = "trends"。

对 EUR/USD 来说，ADF_THRES = 0.60。

也就是说，作者把 ADF 结果当成了一个 市场状态切换器：

更像震荡/回归状态，就启用 reversion 模式
更像延续/方向状态，就启用 trend 模式。

这个设计很值得学，因为它不是死板地永远用一个策略，而是先问：

“现在这段市场更适合追，还是更适合反着做？”
这是很多量化策略从“玩具”走向“能用”的关键一步。这个“ADF 数值具体统计学含义是什么”在仓库里我没有拿到实现文件，因此这里只能确认它被当作 regime switch 使用，不能把它说成更具体的统计结论。

5）趋势模式怎么开仓

趋势模式下，开仓逻辑非常直白：

open_trends_buy: short_sma > long_sma 且 aveSlope > thres
open_trends_sell: short_sma < long_sma 且 aveSlope < -thres。

这说明趋势单不是只看均线金叉死叉，而是 均线方向 + 价格斜率确认 双重过滤。

也就是：

短均线站上长均线，不够
还要最近价格斜率明显朝上，才真买

反之做空同理。

对 EUR/USD，这里趋势开仓阈值 THRES1 = 0.045。趋势离场阈值 THRES2 = 0.023。

这背后的想法是：

不要因为一次弱金叉就贸然追；要等价格“真的在动”。

所以它不是传统慢速日线趋势，而是更偏 tick 级别的短线趋势跟随。

6）均值回归模式怎么开仓

均值回归模式更有意思，因为它和趋势模式几乎是“反着来”的。

代码里：

open_reversion_buy: short_sma < long_sma 且 aveSlope < -thres
open_reversion_sell: short_sma > long_sma 且 aveSlope > thres。

也就是说：

当价格已经明显偏弱、短均线压在长均线下、斜率也在向下时，它反而去找 反弹买点
当价格已经明显偏强、短均线上穿长均线、斜率向上时，它反而去找 回落卖点。

所以这不是“均线回归到均线”的那种慢策略，而更像：

短线冲过头了，做一次向均值弹回。

对 EUR/USD，均值回归的开仓阈值 THRES3 = 0.045，离场阈值 THRES4 = 0.01。

7）平仓逻辑怎么设计

这套代码不是只靠止盈止损硬扛，它还有 信号型平仓。

趋势单平仓
多头：aveSlope < thres2 就关
空头：aveSlope > -thres2 就关。

意思是：
趋势单只要发现“推动力衰减了”，哪怕没打到止盈止损，也可以先走。

均值回归单平仓
回归多单：当 short_sma > long_sma*(1+thres/100) 时平
回归空单：当 short_sma < long_sma*(1-thres/100) 时平。

也就是：
均值回归单并不盯着“斜率衰减”，而是盯着 回归是否已经完成、均线关系是否重新修正。

这说明作者没有把 trend 和 reversion 当成同一种东西处理，而是：

趋势单：看动能衰减
回归单：看偏离修复

这点思路是对的。

8）下单方式：市价单 + 固定止盈止损

buy_send_order 和 sell_send_order 里，下的是 market 单，并且会直接附带 stop loss / take profit。Execution.execute_order() 也是用 OANDA API 直接发市价单。

EUR/USD 参数里：

趋势单：SL1 = 15.0 / 10000，TP1 = 15.0 / 10000
回归单：SL2 = 10.0 / 10000，TP2 = 10.0 / 10000。

也就是大致：

趋势：15 pip 止损 / 15 pip 止盈
回归：10 pip 止损 / 10 pip 止盈。

所以这套系统的收益风险并不是那种“盈亏比巨大、胜率很低”的模式，而更像：

比较均衡的短线型系统，靠入场过滤提高胜率。
这部分是根据参数结构做的归纳。

9）它更像什么风格

如果你问我，这套策略最像什么？

我会说它最像：

“低频一点的 tick 驱动短线系统，而不是现代意义上的微观高频。”

原因很简单：

它用 tick 触发信号，但流动性判断基于 M15 K 线
流动性每 600 秒更新一次
核心还是均线、斜率、ADF 切 regime
没有队列位置、盘口深度、OFI、microprice、maker/taker 控制这些现代微观结构核心元素。

所以它不是那种你常研究的“盘口级微观 edge”系统，更像：

一个事件驱动框架里实现的短线趋势/回归切换策略。

10）这套策略的优点

我觉得它的优点有 4 个。

第一，它不是只会一种逻辑，而是会 切换 regime。这比单一 MA 交叉强很多。

第二，它有 流动性过滤。很多开源策略最缺的就是这一步。

第三，它是 事件驱动 + 多线程 结构，后续扩展实盘比单脚本循环更舒服。

第四，它有 双层退出机制：
既有硬止盈止损，也有信号失效主动平仓。

11）它的缺点和你要特别注意的地方

这套仓库也很明显有一些问题。

第一，技术栈很老。README 明写推荐 Python 2.7、oandapy、MongoDB 3.2.4。

第二，它的参数是强手工调的，泛化能力有限。EUR/USD 参数、AUD/USD 参数、USD/CNH 参数都得分开配。

第三，close_order() 会把该品种当前 trades 全部关掉，所以它更像“单策略单仓位”假设，不适合你后面想做多策略叠加。

第四，它对交易成本和滑点的建模比较粗。市价单、固定 TP/SL，这在真实 FX 环境里很容易受点差和时段影响。这个判断是基于它的执行代码结构得出的。

12）最适合把它当什么来学

我不建议把它当成“直接拿去赚钱的现成策略”。

更适合的定位是：

拿来学习“老派事件驱动外汇系统是怎么搭的”，以及“如何在一个框架里把趋势和均值回归结合起来”。

你可以重点学 3 件事：

线程与事件队列怎么拆
流动性过滤 + regime switch 怎么接到策略里
趋势和回归为什么要用不同的入场、不同的离场。
13）我给这套策略的最终定位

我会把它定义为：

“基于 OANDA 的事件驱动外汇自动交易框架，内置一个流动性过滤 + ADF 状态切换 + MA/斜率决策的短线策略示例。”

不是高频，不是盘口流策略，不是纯趋势，也不是纯均值回归。
它是一个 趋势 / 回归混合切换系统。

可以。PyTrendFollow 这套东西，本质上是一套系统化期货趋势跟随框架：它做的不是什么盘口微观高频，也不是短线剥头皮，而是像很多 managed futures / CTA 一样，靠中期趋势 + carry + 波动率目标仓位管理 + 多品种分散来交易全球期货。仓库 README 直接写明：这是一个“systematic futures trading using trend following”的项目，支持 IB 自动交易、自动下载数据、自动换月、历史回测，并且设计成适合用 Jupyter 做研究。README 还提到它“类似多数 managed futures hedge funds”，并声称历史上大约能做到年化 20% 左右、波动 25% 左右；但它同时也明确写了项目还处于 dev status，使用需自担风险。

1. 一句话先讲清它的核心思路

这套策略不是预测“明天涨还是跌”，而是把每个期货品种都转成一个标准化 forecast（预测分数），再按目标波动率去算该持有多少手。也就是说，它真正交易的是：趋势强弱，而不是单纯价格高低。仓库文档里讲得很直白：先用趋势规则得到 normalized forecast，再把它代入仓位公式，让每个品种按“你允许账户承受多少波动”来定手数；文档甚至直接说，“volatility is the main thing we invest”。代码里 Instrument.position() 也确实是拿 forecast、目标日波动和单品种波动去算理想仓位。

2. 这套仓库默认用什么信号

默认规则并不是一堆复杂因子，而是很经典的两类：EWMAC 趋势 和 carry。在 config/strategy.py.template 里，default_rules = ('ewmac', 'carry')，也就是说默认就是把这两个信号混合起来做。模板里还给了每条规则的 bootstrapped weights，例如 carry、ewmac8、ewmac16、ewmac32、ewmac64 都各有权重。

你可以把它理解成：

ewmac 负责回答：现在有没有趋势，趋势有多强？
carry 负责回答：这个期货结构本身，是更偏多头友好还是空头友好？
最后再把两者加权合成一个总 forecast。这个“合成 forecast 再交易”的结构，是这套系统最核心的骨架。代码里 Instrument.forecasts() 会把各规则结果拼起来，weighted_forecast() 再用权重合成一个最终分数。
3. EWMAC 趋势信号到底是什么

trading/rules.py 里，ewmac 的定义很清楚：先对连续期货价格做一个波动率归一化，然后计算快均线 - 慢均线。具体来说，快线周期是 x，慢线就是 4x；默认用了 4 组参数：8, 16, 32, 64，所以实际上对应的是：

ewmac8：快 8，慢 32
ewmac16：快 16，慢 64
ewmac32：快 32，慢 128
ewmac64：快 64，慢 256。

文档里也专门解释了 moving average crossover：当短周期 EWMA 在长周期 EWMA 上方时，说明处于上升趋势；而且它不希望信号只有“全有或全无”两种，所以改成用两条均线的差值来表示趋势强弱，再做标准化。也就是说，它不是简单的“金叉买、死叉卖”，而是：

均线差越大，趋势分数越高，仓位也就越大。

4. carry 信号在这套系统里怎么做

这套 repo 的 carry 也不是只写了一个概念，而是分成两种实现：

如果这个品种有现货序列，就用 spot vs future 的差来算 carry_spot
如果没有现货序列，就用 当前要交易的合约 vs 下一个换月合约 的价差来算 carry_next。

carry_spot 的逻辑是：
spot - current future，再按到期时间年化，然后做 90 日 EWMA 平滑并标准化。

carry_next 的逻辑更贴近期货实盘：
用当前交易合约和下一张合约的价格差，除以两张合约的月份差，先做 5 日均值平滑，再标准化，最后再做 90 日滚动均值。直觉上讲：

近月比远月更贵，某些情况下可能偏多头 carry
近月比远月更便宜，某些情况下可能偏空头 carry

所以它不只是追价格，还会利用期限结构本身给出的方向信息。

5. 它怎么处理连续合约和换月问题

这是很多期货回测最容易做假的地方，而这套 repo 至少把这个问题正面处理了。Instrument.panama_prices() 里写得很明确：它用的是最简单的 Panama stitching 方法来做连续期货序列。代码注释直接说：

绝对价格会变得“不太有意义”
但日收益和趋势会被保留下来
对这个系统来说已经足够。

这句话其实很重要。它说明这套系统并不依赖“绝对价格的经济含义”，它主要依赖的是：

价格变化
趋势方向
波动率
换月后仍可连续计算的信号。

另外，README 也明确把“automatic rolling of futures contracts”列为功能之一，roll_progression() 则根据 roll day、trade months、roll shift 去生成每天该持有哪个合约。也就是说，它不是回测时一直拿同一张合约死跑，而是有正式的换月框架。

6. 它怎么把 forecast 变成真正仓位

这部分是系统化 CTA 的灵魂。模板里先设了目标年波动率 annual_volatility_target = 0.125，也就是 12.5%，再换算成日波动目标 annual / sqrt(252)。同时模板里还有一个假设资金规模 capital = 500000。

然后 Instrument.position() 的公式本质上是：

仓位 ≈ forecast × 目标日波动金额 ÷ 单品种波动风险

代码里写成：

forecasts * (daily_volatility_target * capital / 10) / return_volatility

最后再四舍五入成整数手数。这里除以 10，是因为 forecast 被标准化到大致 ±10 附近；文档甚至注明这是 legacy 设计。

这一步的思想很典型：

趋势分数大，就多做一点
单品种最近波动大，就少做一点
目标不是“每个品种固定 1 手”，而是让每个品种对账户波动的贡献更可控。
7. 它不是单品种系统，而是组合系统

Portfolio 类说明得很清楚：这是一个由多种 instrument 组成的组合对象，会按 portfolio_weights 去做组合级别的配置。模板里你能看到一大串品种权重，涵盖商品、外汇、利率、股指等，像 crude、corn、gold、nasdaq、us2、us30、yen 等都有单独权重，而且有些品种被直接设成 0，等于暂时不纳入交易。

也就是说，PyTrendFollow 的思路不是“找一个最强品种狠狠干”，而是：

多资产、多市场、多信号，最后做成一个受控波动的组合。
这非常符合传统 CTA 的做法，因为趋势策略真正的稳定性，通常很大程度来自分散化而不是某一个单独神奇信号。

8. 组合层面还会再做一次波动率控制

很多人看到单品种已经按波动率调仓，就以为结束了。但这套系统更进一步，在 accountCurve 里又做了组合层级的 vol_norm()：它会根据组合历史收益的 EWMA 波动，再次缩放整体持仓，且缩放比例被限制在 0 到 1.5 之间。

这是什么意思？

就是即使单品种都按自己的风险定手数了，整个组合如果最近波动突然放大，系统还是会统一把仓位压下来；如果组合波动较低，才允许稍微放大一点。这是一种很典型的 CTA 风控思路：

单品种层：风险平价化
组合层：总波动封顶。
9. 它考虑了成本，不是纯理想化回测

accountCurve.returns() 不是只算持仓盈亏，它还把下面几项都算进去了：

持仓产生的价格变动收益 position_returns
换仓/交易引发的滑点估计 transaction_returns
佣金 commissions
点差 spreads。

其中 transaction_returns 里用了 slippage_multiplier = .5，也就是把交易滑点粗略地建模成过去两天价格变动的一部分。虽然这个模型未必很精细，但至少说明仓库作者知道：

趋势系统如果不扣成本，回测很容易看起来比真实世界漂亮。

10. 这套策略为什么可能有效

因为它抓的是两个在期货里非常经典、而且跨市场长期被研究和实盘使用过的东西：

第一是趋势延续。市场在很多阶段并不会马上均值回归，而是会持续一段时间，尤其是宏观驱动强、资金跟随明显的时候。EWMAC 就是在捕捉这种“已经走起来并且还在延续”的状态。文档里把 moving average crossover 当作趋势捕捉机制来介绍，正是这个逻辑。

第二是carry / 期限结构。很多期货不是简单随机游走，近远月关系本身会反映供需、持仓成本、便利收益、资金利率等信息。carry_next 用当前合约和下一合约的价差去构造信号，就是在利用这种结构特征。

再加上多品种分散和波动率目标，策略就从“某一个信号赌方向”，变成了：

用很多市场里弱相关的趋势和 carry 机会，去堆出一个更平滑的收益曲线。

组合层：多品种权重 + 总波动再归一

研究层：Jupyter 做实验。

对于想学 CTA 的人来说，这比单独看一个“金叉死叉脚本”有价值得多，因为它覆盖了从研究到实盘的大部分骨架。

13. 但它也有几个很明显的局限

第一，仓库自己就说了是 dev status，不是一个你无脑上线就能放心跑的钱印机。

第二，core.utility.py 里对 norm_vol() 和 norm_forecast() 都明确写了 WARNING: lookahead bias。这很关键，说明某些标准化步骤在研究代码里可能引入前视偏差，直接拿来相信回测结果是不安全的。

第三，Panama stitching 虽然保留了回报和趋势，但代码自己也承认绝对价格没意义。这意味着你要把它用于更复杂的价差解释、期权定价或保证金精确模拟时，需要非常小心。

Thought for 1m 33s

可以。jironghuang/trend_following 这个仓库的核心，不是高频，也不是盘口微观结构，而是一个中长期期货趋势跟随研究项目。README 直接写明：它研究的是 futures 上的 trend following，主要比较两条路线：AQR 风格的时间序列动量（TSMOM），以及Continuous Forecasts（CF），后者是作者参考 Robert Carver 框架做的连续预测法。README 还明确说，这更像研究项目而不是工程化项目，错误处理和部署级特性并没有完善。

1. 先说结论：这套策略到底在做什么

一句话概括：

用多种趋势信号给每个期货品种打分，再按波动率目标决定仓位，最后组合成一个多资产趋势跟随系统。

这套 repo 不是只靠一条均线，而是分成两大研究路线：

TSMOM：最经典的“过去一段时间涨了就做多，跌了就做空”
Continuous Forecasts：把多组趋势规则变成连续分数，再加权合成总 forecast，而不是只有 +1 / -1 这种离散信号。
2. 这套项目研究的市场和时间范围

README 里写得比较清楚：作者用的是 Quantopian 提供的 futures 数据，覆盖 Indices、Bonds、Currencies、Commodities 四大类资产，样本一直到 2016 年。研究分成两段：

样本内：1984 到 2010 年 6 月
样本外：2010 年 6 月到 2016 年。

README 的执行摘要还提到一个很重要的结论：
2010 年 6 月以前，不同配置的趋势策略表现都不错，Sharpe 大于 1；2010 年 6 月之后，风险调整后收益显著下降，Sharpe 掉到 0.7 以下。 这说明作者不是在宣传“稳赚”，而是在认真比较趋势策略在不同时代的退化。

3. 这套 repo 的两条主线分别是什么
A. TSMOM：时间序列动量

README 直接说，这条线参考的是 AQR 的 Time Series Momentum 论文。它的思想很简单：

看单个期货过去一段时间的累计涨跌
如果过去表现为正，就顺势做多
如果过去表现为负，就顺势做空。

在代码里，TSMOM_single_instr_monthly_returns() 的实现是：

先把日收益转成月频
用 12 个月 lookback 看价格变化
price_change > 0 则信号为 +1
否则信号为 -1
再按波动率做杠杆调整，目标是单品种 40% 年化波动参考水平。

所以这条路线的本质就是：

最传统的月频趋势跟随 + 波动率目标仓位。

B. Continuous Forecasts：连续预测分数

这一条更像 Carver 风格。不是只输出 “多/空”，而是给出一个连续 forecast 分数，例如从 -20 到 +20。README 也明确说，这是 loosely based on Robert Carver 的框架；但因为数据限制，作者没有把 Carver 书里很重要的 carry roll returns forecast 加进去，并认为这会影响策略效果。

这意味着这个 repo 的 CF 版本，不是完整的 Carver 多因子系统，而是：

以趋势 forecast 为主、缺少 carry forecast 的简化版连续预测框架。

4. Continuous Forecasts 里到底用了哪些信号

代码里主要是两类规则：

EWMAC
Breakout（Donchian channel breakout）。

README 里的初始化示例给出的默认参数是：

ewmac_variations = [8, 16, 32, 64]
breakout_variations = [40, 80, 160, 320]。

也就是说，它不是只看一个周期，而是同时看：

短一点的趋势
中一点的趋势
更长一点的趋势

然后把这些不同周期的规则结果综合起来。这个思路非常典型：
单一趋势周期容易失效，多周期组合更稳健。

5. EWMAC 信号怎么做

util_futures.py 里，raw_ewmac() 定义很直接：

快速 EWMA：Lfast
慢速 EWMA：默认 Lslow = 4 * Lfast
原始信号 = fast_ewma - slow_ewma
再除以 robust_vol_calc(price.diff()) 做波动率归一化。

所以这个 EWMAC 不是单纯看均线交叉，而是看：

快均线和慢均线差值，相对于当前波动有多大。

这很关键。因为如果不除以波动率，不同品种、不同市场、不同年代之间的信号强弱就不可比。经过归一化后，信号才适合放在同一个组合系统里。

6. Breakout 信号怎么做

raw_breakout() 的实现也很标准：

先取 lookback 窗口内的最高价和最低价
算中间值 roll_mean
然后用
40 * (price - roll_mean) / (roll_max - roll_min)
生成 breakout forecast。

这代表什么？

价格越靠近区间上沿，forecast 越偏正
越靠近区间下沿，forecast 越偏负
不是简单“突破了才买”，而是把价格在区间里的位置连续化。

所以它和 EWMAC 一样，也是在输出一个连续趋势强弱值，而不是离散开关。

7. 为什么它叫 Continuous Forecasts

因为代码不是直接拿 raw signal 下单，而是做了两层标准化：

第一层，compute_forecast_scalar() 会给 raw forecast 乘一个缩放因子，让 forecast 的平均绝对值接近 10。第二层，compute_norm_forecast() 再把结果截断在 -20 到 +20 之间。

这就是 Carver 体系里很典型的思路：

各条规则先转成同尺度 forecast
然后再组合
组合后再映射到仓位。

所以 Continuous Forecasts 的本质就是：

信号不是只有做多/做空，而是有“多强、多弱”的连续刻度。

8. 多条规则怎么合成一个最终信号

在 get_combined_forecasts_single_instr() 里，作者把同一个品种的所有 ewmac 和 breakout forecast 拼起来，然后按权重加权求和。若不传权重，默认就是等权。

所以总 forecast 的逻辑就是：

同一品种 = 多周期 EWMAC + 多周期 Breakout 的加权综合。

这比单一规则稳的原因在于：

有些行情均线差更灵
有些行情突破通道更灵
把它们组合起来，可以降低某一条规则失灵时的单点风险。
9. 这些权重不是拍脑袋给的，而是做过优化

这套 repo 有一个比较实战的点：
它不是直接手工写死规则权重，而是对每个品种、在每个 period 里做了一次 forecast 权重优化。

optimize_sharpe_single_instrument_period() 的做法是：

把该品种的各条 ewmac/breakout forecast 拿出来
在某个 bootstrap 样本区间内
用 SLSQP 优化器
在 权重非负、总和等于 1 的约束下
让策略 Sharpe 最大化。

所以这不是“拍脑袋觉得 ewmac16 更好”，而是：

在训练样本里，用 bootstrap + Sharpe 优化去找不同规则的组合权重。

10. 它怎么防止过拟合得太离谱

这里它用的是一种expanding window + bootstrap 的思路。

代码里有：

create_window_index()：把样本按年份块切成 expanding window
extract_boostrap_periods()：在训练段里抽 block bootstrap 样本
create_dictionary_window_n_bootstrap_index()：把窗口和 bootstrap 索引整合起来。

这意味着它并不是一次性拿全样本调参数，而是：

用前面历史做训练
从训练段里抽多个 bootstrap 子样本
每个样本上分别找较优权重
再滚到后面的 out-of-sample 去看表现。

这比“全样本一次性拟合”要严肃得多，虽然仍然不能彻底杜绝过拟合。

11. 仓位管理是这套策略的灵魂之一

这套 repo 很明显不是只研究信号，它非常强调波动率目标仓位。

单品种层面

在 compute_neg_sharpe() 和 TSMOM_single_instr_monthly_returns() 里，都用了相似思想：

先估计波动率
再用 target_vol / realized_vol 算杠杆
波动大就缩仓
波动小就放大。

其中，TSMOM 路线直接用的是：

leverage = 0.4 / vol，也就是 40% 年化参考波动。
组合层面

在 compute_optimal_leverage_all_instruments() 里，作者又做了一层组合控制：

先按单品种参考波动 ind_instr_ref_volatility
再根据所有品种 forecast 满载时的组合波动
算出一个 portf_vol_adj_factor
把组合总风险压到 max_annual_volatility。

README 示例参数里默认是：

max_annual_volatility = 0.15
ind_instr_ref_volatility = 0.4。

这说明这套系统不是简单“每个品种等资金”，而是：

单品种先做风险归一，组合再做总波动上限控制。

12. 它不是天天小调仓，而是设了“再平衡阈值”

这点很值得注意。
代码里有参数 forecast_diff_before_rebal，README 示例默认给的是 6.0；而在回测逻辑里，只有 forecast 变化超过这个阈值，才触发 rebalancing。

这背后的思路非常实际：

forecast 在小范围摆动，不值得频繁交易
只有信号明显变化了，才去改仓
这样可以压低交易成本和换手。

所以这套系统不是“每根 bar 都重新下单”，而是有明显的交易迟钝带。这是中低频趋势系统里很常见、也很重要的设计。

13. 成本也考虑了，但模型比较研究化

README 示例里有：

commission = 20
notion_capital_per_position = 20000
fix_capital = 500000。

在 compute_neg_sharpe() 和 get_commod_returns() 中，作者会在触发再平衡时扣掉 commission。也就是说，这不是完全忽略交易成本的理想化回测。

但它仍然是研究级简化模型，不是交易所级别的完整成本建模。比如：

滑点
合约乘数细节
保证金占用
换月冲击成本

这些都不是这套 repo 的重点。README 也明确说了，它不是工程化部署项目。

14. TSMOM 和 Continuous Forecasts 的差别，你可以这样理解
TSMOM

更像：

单一大方向趋势判断
信号比较朴素
更接近论文原始定义
月频、方向型、离散型。
Continuous Forecasts

更像：

多规则、多周期
连续分数，不是 فقط 多空二值
通过 forecast scaling / capping 统一尺度
再做规则组合、权重优化和组合级风控。

所以如果用一句白话说区别：

TSMOM 像“判断方向”；CF 像“判断方向 + 强弱 + 配仓幅度”。

15. 这套策略为什么在逻辑上可能有效

因为它抓的是趋势跟随最经典的两个来源：

第一层：价格延续

如果一个期货市场在一段时间里持续朝同一方向走，
不管原因是宏观、政策、库存、供需还是避险，
趋势系统都不需要知道原因，它只要识别“是否在持续”。

第二层：风险控制

很多人以为趋势系统的 alpha 只来自信号，其实很大一部分稳定性来自：

波动率目标
分散化
组合风险约束
不过度频繁调仓。

这个 repo 其实把这套东西都做进去了，所以它更接近“完整框架”，而不是一个信号 toy example。

16. 这套 repo 的优点

我会给它几个比较高的评价：

第一，它不是只会讲理论，而是把TSMOM 和 Carver 风格连续 forecast 真正写成了可研究的类。

第二，它不是只做信号，而是连：

规则标准化
forecast 组合
bootstrap 优化
波动率目标
组合总风险控制
再平衡阈值
都覆盖了。

第三，它对趋势退化问题是诚实的。README 明说 2010 年后表现下降，不是那种只晒最漂亮回测图的 repo。

17. 这套 repo 的局限

也很明显。

第一，它缺少 Carver 框架里很重要的 carry roll returns forecast，README 自己都承认这会影响策略效果。

第二，它是研究项目，不是生产系统。README 明确说没有做工程化错误处理和部署特性。

第三，它的数据只到 2016，结论带有明显时代局限。README 对样本范围写得很清楚。

第四，它偏中长期 CTA / futures trend following，并不适合你之前一直在研究的那种盘口微观高频。它不是 order book alpha，也不是 OFI / microprice。

18. 最后给你一个最接地气的总结

jironghuang/trend_following 的核心就是：

在多个期货品种上，用 EWMAC 和 Breakout 这些趋势规则生成 -20 到 +20 的连续 forecast，按权重组合成总信号，再用波动率目标和组合风控把仓位控制住；同时拿 AQR 的 TSMOM 作为另一条基准路线来比较。

所以它属于非常典型的：

中低频、多资产、系统化 CTA 趋势跟随研究框架。

你要是愿意，我下一条可以继续帮你拆成你最熟悉的格式：
“策略名 / 核心信号 / 开仓 / 平仓 / 仓位 / 风控 / 优点 / 缺点 / 适不适合你改造成 IBKR 或日股期货版本”。
